## Core innovations

### Glyph IPC via Windows named file mapping

Native glyph engine (`native_glyph_engine.exe`) communicates with callers through
a Windows named file mapping `Local\GRAMMAP_<tag>`.

**Binary layout (all packed, no alignment padding):**

`IPCHeader` (28 bytes):
| Field | Type | Size | Description |
|-------|------|------|-------------|
| magic | uint64 | 8B | `0x47524D4D525F5447` ("GRAMMR_TG") |
| version | uint32 | 4B | protocol version |
| totalSize | uint32 | 4B | total mapping bytes |
| glyphCount | uint32 | 4B | number of GlyphEntry records |
| status | uint32 | 4B | `0=EMPTY, 1=READY, 2=PROCESSED` |
| reserved | uint32 | 4B | — |

`GlyphEntry` (32 bytes):
| Field | Type | Size | Description |
|-------|------|------|-------------|
| codepoint | uint32 | 4B | Unicode codepoint |
| glyphType | uint16 | 2B | glyph type code (e.g. 0x1001 = NP prototype) |
| pad | uint16 | 2B | — |
| features | uint8[16] | 16B | glyph feature vector |
| reserved1/2 | uint32×2 | 8B | — |

Status flow: caller writes entries → sets status=READY(1) → engine reads + processes → writes result entry after entries → sets status=PROCESSED(2)

**Ported to JS:** `src/ipc/glyph-ipc.js` (layout constants, reader helpers, INT4 program encoder)

### INT4 Lane ISA (glyph_compute.hlsl)

`CS_GlyphExec` (cs_6_0, numthreads(64,1,1)) executes INT4-packed glyph programs:

| Opcode | Nibble | Operation |
|--------|--------|-----------|
| NOP | 0x0 | No-op |
| LOAD | 0x1 | r0 = stateBuffer[lane*4] |
| STORE | 0x2 | stateBuffer[lane*4..2] = r0,r1,acc |
| ADD | 0x3 | r0 = r0 + r1 |
| MUL | 0x4 | r0 = r0 * r1 |
| DOT | 0x5 | acc = r0 * r1 |
| NORM | 0x6 | RMS normalize(r0,r1) |
| EXP | 0x8 | r0 = exp(r0) |
| SUM | 0x9 | acc += r0 |
| MAX | 0xA | r0 = max(r0,r1) |
| MIN | 0xB | r0 = min(r0,r1) |
| MOV | 0xC | r1 = r0 |
| ESC | 0xF | Signal XCFE routing → outBuffer[lane] \|= 0x80000000 |

8 nibbles packed per uint32 (LSB = first nibble). Per-lane state: r0/r1/acc/pc (float32×4 in stateBuffer).

**Note:** `glyph_compute.hlsl` uses `cs_6_0` profile (compiled with DXC, not FXC). This is the native DX12 glyph execution path — distinct from the cs_5_0 training shaders. Intel HD 4600 may not support cs_6_0 without WARP fallback.

### KBC1 program format

KBC1 (K'UHUL Binary Code v1) is a structured MoE execution program:

```
INPUT
repeat N layers:
  LAYERNORM → ATTN_QKV → ATTN_SOFTMAX → ATTN_OUT → MOE_ROUTE → MOE_DISPATCH → MOE_COMBINE
OUTPUT
```

Instruction encoding: `[uint16 op][uint16 flags][uint32[4] args]` = 20 bytes/instruction.

**Ported to JS:** `src/kbc1/kbc1-program.js` (KBC1Program.minimal16LayerMoE(), KBC1Program.fromManifest())

### KUHUL 3D compiler (Python bootstrap)

`kuhul3d_compiler.py` parses line-oriented K3D source to IR JSON:

- `[CommandName arg arg]` → `{ op: "command", name, args[] }`
- `Type name = expr;` → `{ op: "declare", kind: "Tensor|Cluster|Model|Buffer|Shader", name }`
- `dx12 { ... }` → `{ op: "dx12_block", statements[] }`

IR schema: `{ ir_version: "0.1.0", source, dialect: "kuhul3d-v3-bootstrap", ops[] }`

**Ported to JS:** `src/k3d/compiler.js`

### Field system JSON specs

Four physics force field specs (`π_field_v1` schema):

| Spec | Type | Key parameters |
|------|------|---------------|
| `wind_field_spec.json` | wind | direction: vector3, strength: float [0.1,10.0] |
| `navigation_force_spec.json` | navigation | target, strength |
| `attraction_well_spec.json` | attraction | center, radius, strength |
| `scroll_inertia_field_spec.json` | scroll_inertia | damping, snap |

### MoE warp routing shader

`moe_route_warp.hlsl`: One warp (32 threads) computes top-2 expert routing:
- Strided dot products over expert weight rows
- `groupshared float scores[64]` reduction
- Dispatch(1,1,1) — single warp per token

### Glyph token map (kuhul.json)

101 tokens → glyph IDs, including K'UHUL sigils: `⟁Acenter`(1), `⟁Fcol`(0), `⟁Jbetween`(2), `⽽Tran`(3), `⟦FUNC⟧`(4), `⟦LOOP⟧`(5), `⟦VAR⟧`(6), `⟦IF⟧`(7), `⟦ELSE⟧`(8), `⟦RETURN⟧`(9), `⟦KUX⟧`(10), `⟦SHA256⟧`(11).

## Multi-language binding pattern

The native glyph engine is callable from:
- **C++**: direct `process_ipc_mapping()` call
- **Python**: `ctypes.windll.LoadLibrary(...)` + mapped memory
- **Node.js**: FFI or `child_process.spawn(native_glyph_engine.exe, ['--mmap-name', tag])`

This pattern is what the SW-1 bot layer uses to call native engine capabilities.

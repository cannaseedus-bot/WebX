## Key Innovation: Authority-Based Instantiation

The factory enforces a strict policy from `manifest.json`:

```json
{
  "modular_dependency_only": true,
  "execution_authority_requires_explicit_command_binding": true
}
```

**Micronauts are instantiated only through explicit SCX command/program bindings.**  
Directory scanning to discover micronauts is a policy violation — this is the fix for Bug 4 (Supernaut factory hardcoded dead path with tight retry loop).

---

## Design Contract

Registry entry shape:
- `id` — unique string identifier
- `name` — human-readable name
- `binding` — SCX command or program identifier (explicit, not path-derived)
- `version` — semver
- `status` — UNBOUND / BOUND / ACTIVE / MUTATING / RETIRED
- `mutationCount` — cumulative mutations applied

Policy validation: `registry.validatePolicy()` confirms all entries have bindings and no UNBOUND entries remain before any execution authority is granted.

## Merge Target

`src/micronaut/factory.js` — authority contract as JS frozen constants + MicronanutRegistry class.

## Notes

- The `python_bridge.obj` presence confirms the factory was originally paired with the Python training pipeline (adapter mutation engine). This bridge is not portable to JS.
- `bigram_merger.obj` is a training utility — bigram merge table construction for the SCXTOK tokenizer. Not ported here (belongs to SCX.v1.0.0 merge).
- The mutation/evolution pattern (mutation_tracker + evolution_engine) is the precursor to the v3.0.0 reward/mutation/evolution shaders.
